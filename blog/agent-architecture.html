从零理解 AI Agent 架构

2026-07-02 · 8 min read

Agent 这个词在 2025-2026 年被用得太多了，几乎每个 LLM 应用都自称 "Agent"。但真正理解 Agent 架构需要搞清楚几个核心组件。这篇文章是我从嵌入式转 AI Agent 方向过程中的学习总结。

Agent 不是什么

先排除最大的误解：

  • Agent ≠ Chatbot（被动回答 → 主动决策）
  • Agent ≠ Workflow（固定流程 → 动态规划）
  • Agent ≠ 单纯的 LLM 调用（API 包装 → 自主循环）

核心组件

一个真正的 Agent 系统包含五个关键模块：

1. Orchestrator（调度器）

Agent 的"大脑前额叶"。负责接收用户意图，决定下一步做什么：是调用工具、搜索记忆、还是直接回复。它维护着整个对话的 state machine。

class Orchestrator:
    def __init__(self, reasoner, tools, memory):
        self.reasoner = reasoner
        self.tools = tools
        self.memory = memory
        self.state = AgentState.IDLE

    async def run(self, user_input):
        context = await self.memory.recall(user_input)
        while self.state != AgentState.DONE:
            action = await self.reasoner.decide(user_input, context)
            if action.type == "tool_call":
                result = await self.tools.execute(action)
                context += result
            elif action.type == "respond":
                return action.payload

2. Reasoner（推理器）

基于 LLM 的决策引擎。最经典的是 ReAct 循环（Reasoning + Acting）：先想，再做，再观察结果，循环往复。

ReAct 的核心 insight 是：让模型把思考过程显式输出，而不是黑盒式地直接给答案。这样每一步都可追踪、可干预。

3. Tool Registry（工具注册表）

Agent 能调用的一切外部能力的注册中心。每个工具声明自己的 schema（输入/输出格式），Agent 通过 LLM 的 function calling 来调度。

@register_tool(name="search_web", desc="搜索互联网")
async def search_web(query: str, max_results: int = 5):
    """搜索互联网获取最新信息"""
    results = await search_api(query, max_results)
    return {"results": results}

这里的关键是 声明式注册——工具自己描述自己，LLM 根据描述决定何时调用。和传统的 RPC/微服务架构思路完全不同。

4. Memory（记忆系统）

分为三层，和人类认知模型类似：

  • 工作记忆：当前对话上下文（context window）
  • 短期记忆：本次会话的关键信息摘要
  • 长期记忆：向量化存储的历史对话和知识
Memory 是 Agent 跨会话连贯性的关键。没有 Memory 的 Agent 就像金鱼，每次对话从零开始。

5. State Manager（状态管理）

维护 Agent 的生命周期。哪些工具被调用了、当前进度到哪、是否需要人类确认。类似操作系统的进程控制块。

和 Spring Boot 的类比

如果你有后端开发经验，可以这样理解：

  • Orchestrator ≈ DispatcherServlet（中央调度）
  • Reasoner ≈ Service 层业务逻辑（决策核心）
  • Tool Registry ≈ Spring Bean 容器 + AOP（声明式注册和代理调用）
  • Memory ≈ 数据库 + Redis 缓存（分层存储）
  • MCP Protocol ≈ RESTful API 标准（工具间通信协议）

当前主流框架对比

  • LangGraph：图结构编排，适合复杂多步流程，但学习曲线陡
  • CrewAI：多 Agent 协作，角色分工明确，但灵活性受限
  • OpenAI Agents SDK：官方出品，简洁但生态封闭
  • AutoGen：微软出品，对话式多 Agent，代码执行能力强

选型建议：先从单 Agent + 几个工具开始，用 LangGraph 编排复杂流程。不要一开始就上多 Agent，协调成本远超收益。

总结

Agent 架构的本质是 让 LLM 从"被动回答"变成"主动决策"。理解了 Orchestrator-Reasoner-Tool-Memory 这四层，大部分 Agent 框架的设计都能快速看懂。

下一步建议：动手写一个最简 Agent（< 200 行 Python），不要依赖框架，自己实现 ReAct 循环和工具注册。这是理解一切 Agent 框架的基础。
