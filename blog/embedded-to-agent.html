从嵌入式到 AI Agent：一个转行者的笔记

2026-07-02 · 6 min read

做了几年嵌入式（交换机 BSP、蓝牙协议栈），现在转到 AI Agent 方向。这篇文章记录转行过程中的真实感受和经验，给同样想转的人参考。

为什么转

不讲情怀，说真实的三个原因：

  • 天花板：嵌入式岗位的薪资上限和成长空间，说实话在国内市场比较有限。做 BSP 五年和做 BSP 十年的区别不大。
  • 趋势：2025 年 Agent 招聘井喷，大厂都在建团队。供需关系决定了这是一个时间窗口。
  • 可迁移：嵌入式的系统思维（状态机、资源约束、实时性）和 Agent 开发高度可迁移。不是从零开始。

转行地图

我的学习路径（按周排）：

  1. Week 1-2：Python 基础。从 C 背景过来非常快，主要学装饰器、async/await、类型注解。
  2. Week 3：LLM API。OpenAI / GLM 的 chat completion、function calling、streaming。
  3. Week 4：Agent 架构。手写 ReAct 循环、工具注册、记忆系统。
  4. Week 5：RAG。向量数据库（Chroma/FAISS）、embedding、检索策略。
  5. Week 6：LangChain/LlamaIndex 框架。不是为了用，是为了读懂源码。
  6. Week 7+：MCP 协议、多 Agent 编排、LangGraph。做项目。

哪些技能可以直接迁移

如果你的嵌入式中级以上，你已经具备了 Agent 开发需要的 60% 的底层能力。
  • 状态机思维：Agent 的 Orchestrator 本质就是一个状态机。嵌入式中写了无数个。
  • 资源约束意识：嵌入式养成的"内存不是无限的"直觉，在 Agent 里就是 token 管理和 API 限速。
  • 协议解析：MCP 底层是 JSON-RPC，和嵌入式里解析二进制协议包比起来简单多了。
  • 调试能力：嵌入式调试用 JTAG/log，Agent 调试用 trace/日志。方法论一样。
  • 系统思维：从 BSP 到应用层的全栈理解，和从 LLM 到 Tool 到 UI 的全栈理解是同构的。

最大的坑

转行不是技术难，是思维方式要变：

  • 确定性 → 概率性：嵌入式里代码行为是 100% 确定的。Agent 里 LLM 的输出是概率性的。同样的输入可能不同输出。这需要全新的错误处理思路。
  • 性能优化方向不同：嵌入式优化 CPU/内存/功耗。Agent 优化 token 数/延迟/准确率。
  • 测试方式不同：嵌入式写单元测试覆盖边界条件。Agent 需要评估集（eval set）和回归测试。
  • "能跑"不等于"能用"：嵌入式代码跑通了基本就行。Agent 跑通了可能输出一堆垃圾，需要 prompt 工程和评测调优。

面试经验

Agent 岗位面试核心考点（按频率排）：

  1. Agent 架构设计（ReAct vs Plan-and-Execute）
  2. RAG 系统（embedding/chunking/re-ranking）
  3. Function Calling / Tool Use
  4. MCP 协议
  5. Prompt Engineering
  6. 多 Agent 协作
  7. LangChain / LangGraph 源码级理解

加分项：有开源 Agent 项目、有实际落地经验、理解 LLM 训练流程。

给同样想转的人

  • 不要花太多时间学框架 API，框架会变。学原理。
  • 一定要动手做项目，哪怕很小的。一个 200 行的 ReAct Agent 比看完一门课价值大。
  • 简历上不要写"熟悉 LangChain"。写"用 Python 实现了 ReAct Agent 循环，支持工具注册和记忆系统"。
  • 嵌入式背景是你的差异化优势，不是包袱。面试时强调系统思维和底层能力。
