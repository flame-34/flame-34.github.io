MCP 协议详解：Agent 的 USB 标准

2026-07-02 · 10 min read

2024 年底 Anthropic 发布了 MCP（Model Context Protocol），2025-2026 年迅速成为 Agent 生态的事实标准。如果你不理解 MCP，面试 Agent 岗位会被秒挂。这篇文章讲透它。

一句话理解 MCP

MCP 就是 AI Agent 世界的 USB 接口标准。在此之前，每个工具和每个 Agent 之间都需要单独适配，N×M 问题。MCP 之后，工具和 Agent 各自实现一次 MCP 接口，N+M。

为什么需要 MCP

在 MCP 出现之前，给 LLM 接工具是这样的：

  • OpenAI 有 function calling 格式
  • Anthropic 有 tool_use 格式
  • 各种 Agent 框架（LangChain、AutoGen）各有各的工具定义方式
  • 你想把一个搜索工具接到 3 个不同框架上，要写 3 遍适配代码

这就是经典的 N×M 问题。MCP 的目标是标准化这层接口。

MCP 的架构

MCP 是 client-server 架构：

角色谁职责
MCP HostCursor / Claude Desktop / 你的 Agent面向用户的应用
MCP ClientHost 内部的协议层和 Server 通信
MCP Server工具提供方（文件系统、数据库、API）暴露能力

通信方式

  • stdio：本地子进程，标准输入输出通信（最常见）
  • SSE (Server-Sent Events)：远程 HTTP 长连接

MCP Server 能暴露什么

三种能力：

1. Tools（工具）

最常用。让 Agent 能"做"事情。比如：搜索、发邮件、查数据库、执行代码。

{
  "name": "search_db",
  "description": "在用户数据库中搜索记录",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": {"type": "string", "description": "搜索关键词"},
      "limit": {"type": "number", "default": 10}
    },
    "required": ["query"]
  }
}

2. Resources（资源）

让 Agent 能"读"数据。类似 REST API 的 GET。比如：读取文件内容、获取配置、查看日志。

// 资源用 URI 标识
"resources": [
  {"uri": "file:///project/src/main.py", "name": "主入口文件"},
  {"uri": "config://app/settings.json", "name": "应用配置"}
]

3. Prompts（提示模板）

预定义的提示词模板。Server 可以提供一些高质量的 prompt 模板供 Host 使用。

消息格式：JSON-RPC 2.0

MCP 底层用 JSON-RPC 2.0 通信。一个典型的工具调用流程：

// 1. Client → Server: 初始化握手
{"jsonrpc": "2.0", "method": "initialize", "id": 1,
 "params": {"capabilities": {}, "protocolVersion": "2024-11-05"}}

// 2. Server → Client: 返回能力声明
{"jsonrpc": "2.0", "result": {
  "capabilities": {"tools": {}, "resources": {}},
  "protocolVersion": "2024-11-05"
}, "id": 1}

// 3. Client → Server: 请求工具列表
{"jsonrpc": "2.0", "method": "tools/list", "id": 2}

// 4. Server → Client: 返回所有工具
{"jsonrpc": "2.0", "result": {"tools": [...]}, "id": 2}

// 5. Client → Server: 调用工具
{"jsonrpc": "2.0", "method": "tools/call", "id": 3,
 "params": {"name": "search_db", "arguments": {"query": "users"}}}

// 6. Server → Client: 返回结果
{"jsonrpc": "2.0", "result": {
  "content": [{"type": "text", "text": "Found 42 users..."}]
}, "id": 3}

手写一个 MCP Server（Python）

from mcp.server import Server
from mcp.types import Tool, TextContent
import mcp.server.stdio
import asyncio

server = Server("my-tools")

@server.list_tools()
async def list_tools():
    return [Tool(
        name="get_weather",
        description="获取指定城市天气",
        inputSchema={
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "城市名"}
            },
            "required": ["city"]
        }
    )]

@server.call_tool()
async def call_tool(name: str, arguments: dict):
    if name == "get_weather":
        city = arguments["city"]
        # 你的业务逻辑
        weather = f"{city}今天晴，25°C"
        return [TextContent(type="text", text=weather)]

async def main():
    async with mcp.server.stdio.stdio_server() as (r, w):
        await server.run(r, w, server.create_initialization_options())

asyncio.run(main())

把这个 Server 跑起来，任何支持 MCP 的 Host（Claude Desktop、Cursor、你的 Agent）都能直接调用 get_weather 工具，零适配成本。

MCP vs Function Calling

面试高频题。两者不是替代关系：

维度Function CallingMCP
层级LLM API 层（OpenAI/Anthropic 定义）应用协议层（跨 LLM）
作用让 LLM 输出结构化调用意图标准化 Agent ↔ Tool 通信
范围单次 LLM 调用整个工具生态
关系MCP Server 内部可以用 Function Calling 和 LLM 对话

当前生态（2026 年中）

  • Anthropic：Claude Desktop 原生支持，生态最完善
  • OpenAI：ChatGPT/Codex 已支持 MCP Server 接入
  • 开源：已有 200+ 开源 MCP Server（文件系统、GitHub、数据库、浏览器...）
  • 趋势：从 "plugin for X" 走向 "MCP server for anything"

总结

MCP 解决的是 Agent 生态的 标准化问题。可以类比：

  • HTTP 标准化了文档传输
  • RESTful 标准化了 API 设计
  • SQL 标准化了数据库查询
  • MCP 标准化了 Agent ↔ Tool 通信

如果你在准备 Agent 方向面试，MCP 是必考题。理解到 JSON-RPC 消息级别就够了。
